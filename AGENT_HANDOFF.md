# Damulab Admin Mockups — Handoff

Обновлено: 19 апреля 2026

Этот файл нужен для новых чатов/инстансов агента, чтобы не пересказывать контекст по админке заново.

## 2) Принятые продуктовые решения

### Вопросы: ручное добавление
- Для single/multi правильность задается явно:
  - `single` — `radio` (ровно один правильный)
  - `multi` — `checkbox` (несколько правильных)
- Добавление вариантов присутствует прямо в UI: `Добавить вариант` / `Удалить последний`.
- Тип `Сопоставление` сделан в формате 2 колонок + отдельный ключ соответствий (как в пользовательском примере).
- Тип `Ввод своего значения` поддерживает:
  - короткий ответ (число/слово)
  - заполнение пропусков в тексте через шаблон `[[1]]`, `[[2]]` + ключ ответов.

### Вопросы: генерация ИИ
- `Источник` не задается в форме генерации вручную.
- `Источник` показывается в результате у каждого сгенерированного вопроса.
- Есть flow quality-check (после второго прохода LLM), статусы и действия (одобрить/удалить/исправить).

### Темы
- Дерево тем в `topic-tree-editor.html` сделано с разворачиванием веток (`details/summary`).
- Заголовок предметный (например: `Темы: Математика`), а не общий «нулевой уровень».
- Кнопка `Переименовать` показывается после выбора темы.

### Лекции
- В `lecture-editor.html` используется Quill + KaTeX.
- RU/KK переключаются вкладками, а не двумя узкими колонками.
- Есть фильтр по темам.
- Блок контроля понимания содержит 3 режима:
  - `Без контроля`
  - `Автоподбор из банка`
  - `Ручной выбор из банка`
- В списке лекций унифицировано действие: везде `Редактировать`.

## 3) Важные UX/визуальные требования (обязательно для новых агентов)

Новый агент должен сохранять эти правила, если пользователь не попросил иначе:

1. Визуальный язык должен быть близок к текущей админке `admin/index.html`:
- светлая тема
- `Inter`
- slate/indigo палитра
- мягкие карточки и бордеры

2. Макеты должны быть не «картинки», а кликабельные HTML-flow страницы:
- вкладки, переключатели режимов, раскрытия, базовые интеракции JS
- корнер-кейсы вшиваются в flow, а не отдельными «мертвыми» блоками

3. Для форм вопроса всегда явно показывать механизм правильного ответа:
- single: radio
- multi: checkbox
- match: пары/ключ соответствий
- input: ключ ожидаемых значений/пропусков

4. Для лекций не использовать сырой `contenteditable` без редактора.
- Предпочтительно Quill
- Формулы должны рендериться через KaTeX (или совместимый механизм)

5. Не делать слишком «дизайнерский» отрыв от существующего проекта.
- Этот пакет — handoff к реализации (Thymeleaf), а не Figma-концепт.

## 4) Технические заметки

- `lecture-editor.html` содержит fallback: если Quill не загрузился с CDN, показывается textarea.
- Макеты хранятся как статические HTML; перенос в Thymeleaf пока не выполнен.

## 5) Следующие шаги (если продолжать)

1. Разбить макеты на Thymeleaf-фрагменты (`layout`, `sidebar`, `header`, `alerts`, `tables`).
2. Подключить локальные ассеты Quill/KaTeX (без зависимости от CDN).
3. Добавить валидации формы вопросов на уровне JS + подготовить DTO-контракты под backend.
4. Сверить все кнопки/статусы с фактическими API-эндпоинтами проекта.
