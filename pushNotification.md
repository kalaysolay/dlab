# ТЗ: ручное добавление push-уведомлений для учащихся

Статус на 28 апреля 2026: этап 10 дозакрыт в текущем MVP-срезе через `PushProvider` + `StubPushProvider`, DB-backed outbox, админский экран `/admin/push-notifications` и browser/mobile QA ручных сценариев. Production FCM/Web Push/APNs provider и регистрация реальных device tokens остаются отдельным следующим срезом.

## 1) Цель
Сделать админ-экран, где оператор вручную создает push-уведомления для учащихся, указывает серверное время отправки и целевой экран мобильного приложения.

## 2) Scope итерации 1
- Только ручное создание из админки.
- Только отложенная/плановая отправка по времени.
- Два варианта ссылки на экран:
  - `quiz_create_room` (викторина: создание комнаты)
  - `subject_test` (тест по предмету)

## 3) Поля формы
- `text` (строка, обязательно):
  - длина: `1..120` символов (ограничение для мобильного экрана)
  - в UI показать счетчик символов
- `scheduled_at` (дата+время, обязательно):
  - ввод в формате `YYYY-MM-DD HH:mm`
  - интерпретация только как серверное время
- `target_screen` (enum, обязательно):
  - `quiz_create_room`
  - `subject_test`
- `target_payload` (object, условно):
  - для `subject_test` передавать `subject_id`

## 4) Бизнес-правила
- Нельзя сохранить уведомление с пустым текстом.
- Нельзя сохранить уведомление на прошедшее серверное время.
- Push должен быть доставлен в мобильное приложение учащегося в указанное серверное время.
- До отправки уведомление можно редактировать или отменять.
- После отправки уведомление только для чтения (детали/лог статуса).

## 5) UI/UX требования
- Экран админки:
  - форма создания уведомления
  - предпросмотр мобильного пуша
  - таблица запланированных/отправленных уведомлений
- Для времени явно показывать подпись `Серверное время (UTC+X)`.
- Для текста показывать live-счетчик `N/120`.
- Для `target_screen=subject_test` показывать дополнительное поле выбора предмета.
- Страница админки должна быть читаемой на desktop и мобильных ширинах 390/360px без page-level horizontal overflow; таблица и inline edit на mobile отображаются в stacked layout.

## 6) Минимальный API (черновик)
### POST `/api/admin/push-notifications`
Создать уведомление.

Пример тела:
```json
{
  "text": "Через 15 минут стартует викторина. Успей подключиться!",
  "scheduled_at": "2026-04-23 19:30",
  "target_screen": "quiz_create_room",
  "target_payload": {}
}
```

### GET `/api/admin/push-notifications`
Список уведомлений со статусами: `scheduled | sent | cancelled | failed`.

### PATCH `/api/admin/push-notifications/{id}`
Редактирование до отправки.

### POST `/api/admin/push-notifications/{id}/cancel`
Отмена до отправки.

## 7) Критерии приемки
- Админ может создать push с валидным текстом, временем и экраном.
- Уведомление видно в списке со статусом `scheduled`.
- В момент `scheduled_at` (по серверным часам) push отправляется в мобильное приложение.
- Для `subject_test` в payload передается выбранный `subject_id`.
- Сообщения длиннее 120 символов не сохраняются.
- Уведомление на прошедшее время не создается.
