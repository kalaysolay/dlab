# Damulab.kz: agent workflow and handoff protocol

Статус: обязательный стартовый контекст для новых инженерных чатов/агентов.
Дата подготовки: 26 апреля 2026.

Цель документа - не терять контекст между чатами и не позволять агентам расходиться в scope, архитектуре и принятых решениях.

## 1. Главные документы проекта

Перед началом любой реализации агент должен прочитать минимум:

1. `01_SYSTEM_SPECIFICATION.md` - рабочее ТЗ, границы MVP, роли, модули, API верхнего уровня.
2. `02_IMPLEMENTATION_PLAN.md` - порядок реализации по этапам.
3. `03_ARCHITECTURE_VISION.md` - архитектурные решения, слои, модульный монолит, ограничения.
4. `Damulab_Vision_Scope.md` - продуктовое видение и долгосрочный scope.

Дополнительные документы по контексту:

- `ADMIN_USE_CASES.md` - сценарии админки.
- `AGENT_HANDOFF.md` - handoff по уже созданным admin mockups.
- `pushNotification.md` - требования к ручным push-уведомлениям.
- `docs/AI_SAFETY_BASELINE.md` - обязательные privacy/safety guardrails перед задачами AI Content Factory, AI-разборов, AI-тьюторинга или интеграции внешних AI-провайдеров.
- HTML-макеты в корне проекта - пользовательский портал, тесты, аналитика, quiz.
- `admin/mockups/*` - целевые admin UI-flow макеты.

## 2. Принятые решения, которые нельзя переоткрывать без причины

- MVP строится вокруг базового учебного цикла: контент -> тест -> результат -> аналитика -> родитель.
- Backend: Java 21, Spring Boot 3.x, модульный монолит.
- DB: PostgreSQL, миграции через Flyway.
- UI MVP: Thymeleaf templates + compiled CSS + легкий vanilla JS/Alpine, без отдельного SPA.
- Проверка ответов и бизнес-правила выполняются на backend.
- Правильные ответы нельзя отдавать клиенту до завершения теста.
- AI-контент не публикуется без human review.
- Полноценная AI Content Factory, Arena real-time, streak/достижения, LMS checkpoints и advanced content health - post-MVP, если задача явно не говорит обратное.
- Для AI-задач внешним провайдерам нельзя отправлять персональные данные или прямые идентификаторы; результаты тестов, связанные с конкретным учеником, считаются персональными данными внутри Damulab и перед AI должны быть минимизированы/обезличены.
- AI-результаты являются черновиками и не публикуются без human review методистом.
- Email confirmation, password reset через email и email-рассылки не входят в MVP до настройки SMTP.
- Email в MVP используется как логин и не требует подтверждения.
- RU/KK обязательны; казахский язык используется на кириллице.

## 3. Стартовый промпт для нового агента

Скопировать в новый чат перед задачей:

```text
Ты работаешь над Damulab.kz в папке проекта.

Перед реализацией прочитай:
- 01_SYSTEM_SPECIFICATION.md
- 02_IMPLEMENTATION_PLAN.md
- 03_ARCHITECTURE_VISION.md
- Damulab_Vision_Scope.md
- AGENT_WORKFLOW.md

Если задача касается админки, также прочитай:
- ADMIN_USE_CASES.md
- AGENT_HANDOFF.md
- relevant files in admin/mockups/

Если задача касается push, также прочитай:
- pushNotification.md

Если задача касается AI, также прочитай:
- docs/AI_SAFETY_BASELINE.md

Ключевые правила:
- MVP: content -> test -> result -> analytics -> parent.
- Не добавляй email confirmation в MVP.
- Не делай отдельный SPA для MVP.
- Не реализуй AI/Arena/streak/LMS checkpoints как блокеры MVP, если задача явно не просит.
- Не отправляй персональные данные или прямые идентификаторы внешним AI-провайдерам; AI-результат не публикуется без проверки методистом.
- Сохраняй серверную проверку ответов.
- Обновляй документацию и gap report после своей работы.

Перед изменениями коротко сформулируй, какой этап плана закрываешь.
После изменений оставь отчет в формате из AGENT_WORKFLOW.md.
```

## 4. Формат финального отчета агента

Каждый агент в конце задачи должен дать отчет:

```text
## Agent Report

Задача:
- <кратко, что делал>

Измененные файлы:
- <path> - <что изменено>

Что реализовано:
- <фактическое поведение/модули>

Проверка:
- <команды, тесты, ручная проверка>
- <если не запускалось, почему>

Риски:
- <технические/продуктовые риски>

Gap Report:
- <что осталось незакрытым относительно ТЗ/плана>

Решения/допущения:
- <что агент решил сам и почему>

Следующий шаг:
- <самый логичный следующий шаг>
```

## 5. Где хранить накопительный handoff

После каждой существенной задачи агент должен обновить `AGENT_PROGRESS_LOG.md`.

Если файла нет, создать его с такой структурой:

```markdown
# Damulab.kz: progress log and gap report

## Current State

- <краткое состояние проекта>

## Completed Work

### YYYY-MM-DD - <task title>

- Agent/task:
- Files changed:
- Implemented:
- Verification:
- Risks:
- Gaps:
- Next:

## Open Gaps

- [ ] <gap>

## Decisions

- YYYY-MM-DD: <decision>

## Risks

- <risk and mitigation>
```

Правила:

- Не превращать progress log в длинный дневник. Писать только то, что поможет следующему агенту.
- Если gap закрыт, отметить это в `Open Gaps` или удалить после переноса в `Completed Work`.
- Если принято решение, которое меняет MVP/scope/архитектуру, обновить не только progress log, но и профильный документ: `01_SYSTEM_SPECIFICATION.md`, `02_IMPLEMENTATION_PLAN.md` или `03_ARCHITECTURE_VISION.md`.

## 6. Перед началом новой задачи

Агент должен:

1. Прочитать `AGENT_PROGRESS_LOG.md`, если он существует.
2. Проверить, не противоречит ли задача текущим границам MVP.
3. Найти ближайший этап в `02_IMPLEMENTATION_PLAN.md`.
4. Сформулировать короткий план.
5. Работать вертикально: backend, миграции, UI, тесты и документация в разумном scope задачи.

## 7. Definition of Done для агентской задачи

Задача считается готовой, если:

- Изменения соответствуют текущему MVP scope.
- Нет возврата email confirmation в MVP.
- Бизнес-правила находятся на backend.
- Если менялась модель данных, есть миграция.
- Если менялось поведение, есть тесты или явно описана причина их отсутствия.
- UI использует общий layout/паттерны и не дублирует крупные фрагменты без необходимости.
- Ошибки и edge cases видимы пользователю.
- Обновлен `AGENT_PROGRESS_LOG.md`.
- Финальный отчет содержит risks и gap report.
